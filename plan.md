# MAD Backend and Scraping System Plan

## Goal

We want to redesign the MAD scraping and backend system so it is easier to scale, easier to debug, and safer for the web app.

Right now, raw scraping, cleanup, enrichment, deduping, and final app tables are too mixed together. The new plan is to separate the system into clear layers so each part has one job.

## Simple Version of the Plan

The new MAD system should work like this:

`scrapers -> raw tables -> cleaned source tables -> canonical core tables -> enrichment results -> final serving tables -> web app APIs`

This means we stop writing directly into final app-facing tables during scraping and enrichment. Instead, we build final tables from trusted intermediate layers.

## 1. Raw Data Layer

Every scraper or import job should first write data into raw tables.

Examples:

- raw Zillow data
- raw RE/MAX data
- raw KW data
- raw Maps teams data
- raw Realtor data

These raw tables should store:

- original source payload
- source record ID
- scrape URL
- scrape timestamp
- ingestion timestamp
- pipeline run ID
- checksum or fingerprint

Purpose:

- keep original source truth
- allow reprocessing later
- avoid losing data because of bad transforms

## 2. Cleaned Source Layer

After raw ingestion, each source should be cleaned into a normalized source table.

This layer should standardize things like:

- names
- phone numbers
- emails
- websites
- brokerage text
- city/state/zip
- team names

At this stage, the data is cleaner, but still source-specific.

Purpose:

- make different source feeds easier to compare
- keep source cleanup separate from final merging

## 3. Canonical Core Layer

This is the main truth layer.

Here the system decides:

- which rows are the same agent
- which rows belong to the same team
- which brokerage is the correct one
- what internal IDs should be used

Core tables should include:

- `agent_core`
- `team_core`
- `brokerage_core`
- optional `office_core`

This layer should also create proper relationships:

- agent to team
- team to brokerage
- team to office

Purpose:

- replace today’s loose matching logic
- make deduping explicit
- create a stable source of truth for the whole system

## 4. Enrichment Layer

Once canonical agent and team records exist, enrichment systems should run on top of them.

This includes:

- team-size webhook
- CRM agent
- social-media agent
- future review/rating/customer-match enrichments

Important rule:

These enrichments should not directly update the final app-facing tables.

Instead, they should write results into separate enrichment result tables, for example:

- `team_size_runs`
- `tech_stack_runs`
- `social_profile_runs`
- `rating_review_runs`

Each enrichment result should store:

- entity ID
- result payload
- confidence
- status
- source/evidence
- failure reason
- completed timestamp

Purpose:

- keep enrichment history
- rerun enrichments safely
- prevent failed runs from damaging good data

## 5. Serving Layer

This is the layer that prepares data for the MAD web app and backend APIs.

The final app-facing tables should be built from:

- canonical core tables
- latest accepted enrichment results

This layer should produce:

- `mart_agents`
- `mart_teams`
- `mart_dashboard_snapshots`

These can later power or replace:

- `new_unified_agents`
- `Team_Summary`
- `mad_dashboard`

Purpose:

- make app-facing data stable
- prevent raw workflows from mutating app tables directly
- make dashboard and API outputs reproducible

## 6. API Layer

The MAD web app and APIs should read only from the serving layer.

They should not read directly from:

- raw tables
- source-normalized tables
- enrichment run tables
- partially processed tables

Purpose:

- simpler API logic
- safer frontend behavior
- easier caching and scaling

## How the Three Final MAD Tables Should Be Treated

## `new_unified_agents`

This should become the final agent-serving table.

It should contain:

- canonical agent identity
- best contact and location data
- team and brokerage link
- latest enrichment values
- data quality and freshness metadata

It should no longer be treated as a working table where multiple workflows directly write conflicting values.

## `Team_Summary`

This should become the final team-serving table.

It should be built from:

- canonical team records
- team-level enrichment outputs
- latest accepted website, CRM, marketing, and other team data

It should no longer depend on fragile source-specific updates as the main truth.

## `mad_dashboard`

This should become a proper snapshot table.

It should be generated by a scheduled job from trusted serving/core data.

Each snapshot should store:

- snapshot date
- pipeline run ID
- created timestamp
- metric counts
- computation version

Purpose:

- make dashboard metrics auditable
- avoid mixed field types and unclear semantics

## How the New End-to-End Flow Works

### Old style

`scrape -> workflow -> directly update final table -> more workflows -> app reads same table`

### New style

`scrape -> raw table -> cleaned source table -> canonical entity -> enrichment result -> serving table -> app/API`

This is the biggest system change.

## Role of the Existing Three Major Steps

## Team Size Webhook

This becomes an enrichment service that returns:

- team size
- team category
- homepage/team page
- brokerage
- team name
- CRM hints

It should write to enrichment result tables, not directly to final serving tables.

## CRM Agent

This becomes a tech-stack enrichment step that detects:

- CRM tools
- marketing tools

It should also write structured enrichment results first, then those results should be promoted into final serving tables.

## Social Media Agent

This becomes a social-profile enrichment step that finds:

- Facebook
- Instagram
- LinkedIn
- YouTube
- TikTok
- X/Twitter
- score and label

Again, it should write enrichment results first, not mutate final truth directly.

## Why This Design Is Better

This design makes the system:

- easier to debug
- easier to backfill
- easier to scale
- safer for the web app
- better for data lineage
- better for freshness tracking
- less fragile when one workflow fails

It also makes it much easier to answer questions like:

- where did this value come from?
- when was it last updated?
- which source won during dedupe?
- why is this row in the app?
- why is this metric in the dashboard?

## Practical Outcome

After this redesign:

- scrapers only scrape
- normalizers only clean data
- identity resolution only merges and links entities
- enrichments only produce enrichment results
- serving jobs only build final app tables
- APIs only read final serving tables

That clean separation is what will make MAD robust at scale.

## One-Line Summary

MAD should be rebuilt as a layered data platform where raw data is captured first, entities are resolved in the middle, enrichments are tracked separately, and final tables for the web app are generated from trusted serving layers.
